0.2 Architecture commitment — which OFFBOARD sub-mode
Three options, pick one and stick with it:
A. Position-mode setpoint (setpoint_position/local) — easiest. NMPC outputs the next position waypoint. PX4 does everything else. Loses some of the point of NMPC because PX4 re-smooths your setpoint. Use this if you want to ship fast.
B. Attitude setpoint (setpoint_raw/local with type_mask for attitude) — medium. NMPC outputs quaternion + collective thrust at 50–100 Hz. PX4 attitude+rate loop tracks it. This is what the published acados+PX4 NMPC packages use. Default recommendation.
C. Body rate setpoint — hardest. NMPC outputs body rates + thrust. Only PX4 rate loop is in play. Better tracking but more work.
I'd commit to B (attitude setpoint) unless there's a reason otherwise. It's the standard for published NMPC+PX4 work, and the reference implementation (evannsmc/nmpc_acados_px4) uses it.
0.3 NMPC software stack
    • Solver: acados. It's the standard. casadi is the modelling layer, acados is the codegen + RTI solver. Both come together. 
    • Language for NMPC node: C++ for production-style, Python for fast iteration. I'd recommend Python for the prototype, port to C++ only if solve time is borderline. acados has good Python bindings and the solver itself is C anyway. 
    • ROS: ROS 1 Noetic per CLAUDE.md. 
0.4 What "observability" actually means here
This is the conceptual heart of the contribution and it's where most of your novelty work happens. There are at least three things you could mean:
(i) Local-point-cloud spectral spread: take the current LiDAR scan, compute the 3×3 covariance matrix of point positions in the robot frame, look at eigenvalues. Small eigenvalue in some direction = few points spread in that direction = future motion in that direction will be poorly constrained by scan matching. This is what I'd start with. It's computed once per scan, cheap, well-defined.
(ii) Registration information matrix: from FAST-LIO's iEKF, the actual information matrix of the LiDAR factor. This is the right thing mathematically but requires modifying FAST-LIO to expose it. More work, more correct.
(iii) Predicted observability at future poses: this is what perception-aware NMPC needs in its cost — not "observability now" but "observability if I move to position $x_k$ in $k$ steps." You build a local observability field and look it up at each NMPC stage.
My recommendation: start with (i) as the field source — i.e., use the current scan to estimate a local observability field over a coarse voxel grid around the robot, and look up that field over the NMPC horizon. If reviewers push back on rigor, you have a path to (ii). Don't promise (ii) in the abstract.

1.3 System identification for the NMPC model
The NMPC needs an inertia matrix, mass, and thrust constants for your S500.
    • Mass: measure. Include the LiDAR, battery, all mounted gear. Probably 1.4–1.8 kg. 
    • Hover thrust: you already know from flight 29 — HTE = 0.196 normalized. From that you can back out actual hover thrust in Newtons: $T_{hover} = m \cdot g$, normalized thrust 0.196 maps to that. Solve for max thrust per motor. 
    • Moment of inertia: hardest. Three options ranked by accuracy: 
        1. CAD model with motor/battery/LiDAR masses — quick, accurate to ~20% 
        2. Bifilar pendulum experiment — accurate to ~5%, takes a day 
        3. System ID from flight data — accurate but requires excitation flights 
      Start with (1) for the prototype. Refine with (3) if NMPC tracking is poor. 
    • Thrust constant $k_T$ (motor command → thrust): with HTE = 0.196 and hover thrust known, this falls out. Document the assumption that thrust is quadratic in motor command (standard, true for small drones). 
Record all these in CLAUDE.md as you go. Future-you will need them.
1.4 First simulation flight

    • No oscillation when the drone is at the setpoint 
If any of these fail, fix in sim before flying real. Don't proceed otherwise.

Phase 2 — Build the observability monitor
Separate from NMPC. This is a standalone ROS node that subscribes to the LiDAR point cloud and publishes an observability field.
2.1 Define the observability metric precisely
Pick one definition and commit. I'd start with:
For a LiDAR scan with $N$ points ${p_i}$ in robot body frame, the directional observability in direction $\hat{d}$ is:
$$o(\hat{d}) = \frac{1}{N} \sum_i w_i \cdot |\hat{n}_i \cdot \hat{d}|$$
where $\hat{n}_i$ is the surface normal at point $p_i$ and $w_i$ is a weight (range-dependent or constant). Intuitively: directions in which many surfaces have aligned normals are well-constrained for ICP/scan-matching; directions perpendicular to all visible surfaces are degenerate.
This is the standard scan-matching observability from the Zhang & Singh 2016 paper ("On Degeneracy of Optimization-based State Estimators"), well-cited.
For an even simpler first pass: compute the 3×3 covariance matrix of the point positions, take its eigendecomposition, and use the smallest eigenvalue as a scalar degeneracy score. This is cruder but takes 10 lines of code and is enough to validate the pipeline.
Start with the eigenvalue version. Upgrade to the normal-based version if and only if the simpler one isn't sensitive enough in your corridor experiment. This is a real DECIDE LATER point.
2.2 From scalar to field
The observability of the current scan tells you about the current pose. NMPC needs predicted observability at future poses. Two approaches:
Approach A (lazy, recommended first): assume observability is approximately constant over the NMPC prediction horizon (typically 1–2 s at 50–100 Hz). Use one scalar value, look it up the same way at every stage of the NMPC. Justification: at 0.5 m/s and a 1 s horizon, the drone moves 0.5 m, and indoor LiDAR observability doesn't change drastically over 0.5 m. This is a defensible simplification.
Approach B (real perception-aware): maintain a coarse 3D voxel grid over the local environment (say 5×5×3 m at 0.5 m resolution). For each voxel, store an estimate of "what would the observability be if I were standing here?" Update from LiDAR scans as you fly. NMPC looks up the voxel each stage visits. This is much more like the published PAMPC / HPA-MPC formulations.
Start with A. Switch to B in the paper extension section, or as a v2 if A's results are weak.
2.3 Implementation notes
    • Use pcl_ros (already in your dependencies probably) for point cloud handling. 
    • Downsample the input cloud heavily before eigen-decomposition. Hesai JT128 produces ~150k points/scan; you want ~5k for stable PCA. Voxel grid filter at 0.1 m resolution. 
    • Compute normals only if you go beyond Approach A — they're expensive. 
    • Publish the observability metric as a std_msgs/Float32 (for Approach A) or as a small voxel grid message (for B). 
2.4 Sanity test before integration
Drive the drone around in your test environment (manually, no NMPC). Subscribe to the observability topic. Verify:
    • Walking through your corridor produces a visibly different observability value vs being in open space 
    • The transition is smooth, not glitchy 
    • Values are stable when stationary 
If you can't see a clear difference in the metric between open and corridor, your metric is wrong and the whole paper has no signal. Fix this before anything else.

Phase 3 — Couple observability into NMPC cost
Now the actual contribution. Take the working NMPC, add the observability cost, validate it changes behavior.
3.1 Cost formulation
Augment the standard NLS cost with a perception term. In acados Python, this means adding a soft cost expression to your model.
For Approach A (scalar observability):
$$J = \underbrace{\sum_k |x_k - x_k^{ref}|^2_Q}{\text{tracking}} + \underbrace{\sum_k |u_k|^2_R}{\text{effort}} + \underbrace{w_{obs} \cdot \sum_k \frac{1}{o_{current} + \epsilon} \cdot v_k^T v_k}_{\text{observability}}$$
The third term: when observability is low ($o$ small), the cost of high velocity ($v^T v$) goes up. NMPC will naturally slow down to reduce velocity-squared cost when perception is weak. When observability is high, the term vanishes and NMPC tracks normally.
The $\epsilon$ prevents divide-by-zero. $w_{obs}$ is the tuning knob.
This is the simplest possible observability coupling and it makes a defensible IECON contribution. Don't over-engineer it at first.
3.2 Implementing in acados
acados doesn't dynamically change cost weights mid-solve easily. The cleanest way: pass observability as a parameter to the solver. In acados_template, model parameters can be set per-solve and per-stage.
Workflow each solve cycle:
    1. Read latest observability scalar 
    2. solver.set(stage, "p", [observability_value, ...]) for each stage 0..N 
    3. Solve 
Cost expression in the model includes p[0] as a multiplier on the velocity penalty.
3.3 Tuning the weight
This is sensitive. Strategy:
    • $w_{obs} = 0$ recovers standard NMPC (your baseline) 
    • Start with $w_{obs}$ such that, at the worst observed degeneracy, the effective velocity penalty roughly doubles 
    • Sweep in sim across $w_{obs} \in {0, 0.1, 1, 10}$ on the same trajectory 
You will get a clear "too small (no effect), about right (slows in corridor), too large (drone refuses to move)" curve. The sweep is itself a paper figure.
3.4 First test in SITL
In SITL, you don't have a real LiDAR producing variable observability. Two options:
    • Mock the observability publisher: a small script that publishes synthetic observability that drops when the simulated drone enters a defined region. This validates the NMPC machinery without depending on the real perception pipeline. 
    • Run a fake-corridor Gazebo world: harder, but reproduces the full pipeline in sim. 
Start with the mock publisher. Confirm NMPC behavior changes as expected when observability drops. Then move to real.
3.5 First real flight with observability cost
Same trajectory, same test space (your cardboard corridor). Compare:
    • NMPC baseline (Phase 1) → drone tracks aggressively through corridor, possibly drifts 
    • NMPC + observability cost → drone slows automatically in corridor 
Capture full logs. This is your money shot.

4.3 Failure conditions to document honestly
Things that will go wrong and you should document:
    • NMPC solve occasionally exceeding deadline → fallback behavior (last valid solution? abort?). Define and document. 
    • Observability metric briefly glitching → smoothing approach (median over last 5 scans?) 
    • Corridor exit transient — drone suddenly speeds up. Show this and discuss. 
Papers that admit limitations get accepted more often than papers that pretend everything works.
